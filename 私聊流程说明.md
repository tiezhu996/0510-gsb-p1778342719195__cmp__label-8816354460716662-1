# 用户私聊完整流程说明

## 一、系统架构概览

本系统采用 **FastAPI + React + WebSocket** 的前后端分离架构：

- **后端**：FastAPI (Python)，提供 REST API + WebSocket 实时通信
- **前端**：React + TypeScript + Vite，通过 Axios 调用 REST API，通过 WebSocket 实现实时消息
- **数据库**：SQLAlchemy ORM + SQLite，核心表为 `private_messages`
- **实时通信**：WebSocket 单例管理器 `ConnectionManager`，维护用户连接映射

## 三、分步骤详细说明

### 阶段1：建立 WebSocket 连接

#### 前端操作

1. 用户登录后，`AuthContext` 持有 JWT Token 和用户信息
2. `WebSocketProvider` 检测到用户已登录（`isAuthenticated && user`），自动调用 `wsService.connect(user.id)`
3. `WebSocketService.connect()` 构造连接地址：`ws://host:8000/ws/{userId}?token={jwt}`
4. 创建浏览器 WebSocket 实例，注册 `onopen/onmessage/onerror/onclose` 回调
5. 连接成功后启动心跳机制（每30秒发送 `ping`）

**关键代码**：
- 前端：`frontend/src/contexts/WebSocketContext.tsx:38-63` — 自动连接逻辑
- 前端：`frontend/src/services/websocket.ts:45-106` — 连接建立与重连
- 后端：`backend/app/main.py:158-166` — WebSocket 路由注册

#### 后端操作

1. FastAPI 收到 WebSocket 升级请求
2. `websocket_endpoint()` 先 `accept()` 连接
3. 调用 `verify_websocket_token(token)` 验证 JWT：
   - 解码 Token 获取 `sub`（用户ID）
   - 比对 URL 中的 `user_id` 与 Token 中的是否一致
4. 验证通过后，将连接注册到 `ConnectionManager.active_connections[user_id] = websocket`
5. 发送连接确认消息：`{type: "connection", status: "connected", user_id: ...}`
6. 进入消息循环 `while True: await websocket.receive_text()`

**关键代码**：
- 后端：`backend/app/websocket/handlers.py:63-163` — WebSocket 端点处理
- 后端：`backend/app/websocket/connection_manager.py:24-35` — 连接注册

---

### 阶段2：发送私聊消息

#### 步骤 A：前端发送消息

1. 用户在 `PrivateChatPage` 输入消息，点击发送
2. `handleSendMessage(content)` 被调用
3. 检查 WebSocket 是否已连接（`isConnected`）
4. 调用 `sendPrivateMessage(receiverId, content)`
5. `WebSocketContext.sendPrivateMessage` → `wsService.sendPrivateMessage(receiverId, content)`
6. `WebSocketService.sendPrivateMessage()` 构造消息体并通过 `ws.send()` 发送：

```json
{
  "type": "private_message",
  "receiver_id": 2,
  "content": "你好",
  "message_type": "text"
}
```

**关键代码**：
- 前端：`frontend/src/pages/PrivateChatPage.tsx:137-144` — 发送触发
- 前端：`frontend/src/services/websocket.ts:155-162` — 消息构造与发送

#### 步骤 B：后端接收与处理

1. `websocket_endpoint()` 的消息循环收到文本数据
2. `json.loads(data)` 解析消息，判断 `message_type == "private_message"`
3. 调用 `handle_private_message(db, user_id, message)`

**关键代码**：
- 后端：`backend/app/websocket/handlers.py:110-123` — 消息类型路由

#### 步骤 C：消息持久化

1. `handle_private_message()` 提取 `receiver_id`、`content`、`message_type`
2. 验证参数完整性（`receiver_id` 和 `content` 不能为空）
3. 调用 `chat_service.create_private_message(db, sender_id, receiver_id, content, message_type)`
4. Service 层验证：
   - 接收者用户是否存在（查询 `users` 表）
   - 发送者不能是接收者（防止自发自收）
5. 创建 `PrivateMessage` ORM 对象，设置 `is_read=False`
6. `db.add()` + `db.commit()` + `db.refresh()` 写入数据库并获取完整对象（含自动生成的 `id`、`created_at`）

**关键代码**：
- 后端：`backend/app/websocket/handlers.py:224-281` — 私聊消息处理
- 后端：`backend/app/services/chat_service.py:482-530` — 消息持久化
- 后端：`backend/app/models/chat.py:74-97` — PrivateMessage 数据模型

#### 步骤 D：WebSocket 实时推送

1. `db.refresh(private_message)` 加载 `sender` 和 `receiver` 关系
2. 构建响应消息体（包含完整的消息信息）：
```json
{
  "type": "private_message",
  "id": 42,
  "sender_id": 1,
  "receiver_id": 2,
  "content": "你好",
  "message_type": "text",
  "is_read": false,
  "created_at": "2026-05-10T10:30:00",
  "sender_username": "Alice",
  "receiver_username": "Bob"
}
```
3. **推送给接收者**：`manager.send_personal_message(response_message, receiver_id)`
   - 检查接收者是否在线（`receiver_id in active_connections`）
   - 在线则通过 WebSocket 发送 JSON
   - 不在线则跳过（消息已持久化，接收者上线后可通过 REST API 拉取）
4. **推送给发送者**：`manager.send_personal_message(response_message, user_id)`
   - 发送者也需要收到回显，以实时看到自己发送的消息

**关键代码**：
- 后端：`backend/app/websocket/handlers.py:246-269` — 消息推送逻辑
- 后端：`backend/app/websocket/connection_manager.py:67-85` — 个人消息发送

#### 步骤 E：前端接收消息

1. 浏览器 WebSocket `onmessage` 触发
2. `WebSocketService.handleMessage()` 被调用，遍历所有 `messageHandlers`
3. `PrivateChatPage` 中注册的 `onMessage` 回调被触发
4. 判断 `message.type === 'private_message'`
5. 进一步判断消息是否属于当前对话（`sender_id` 和 `receiver_id` 与当前聊天对象匹配）
6. 将消息转换为 `PrivateMessage` 对象，追加到 `messages` 状态数组
7. React 自动重新渲染 `ChatMessageList`，新消息出现在聊天窗口

**关键代码**：
- 前端：`frontend/src/pages/PrivateChatPage.tsx:44-84` — 消息接收与过滤

### 阶段3：标记已读

#### 触发方式1：收到消息时自动标记

1. `PrivateChatPage` 的 `onMessage` 回调中，检测到 `message.receiver_id === user.id && !message.is_read`
2. 自动调用 `markMessageRead(message.id)`（异步，不阻塞渲染）

**关键代码**：
- 前端：`frontend/src/pages/PrivateChatPage.tsx:74-78`

#### 触发方式2：打开页面时批量标记

1. `PrivateChatPage` 有一个独立的 `useEffect`，监听 `messages` 变化
2. 过滤出所有 `receiver_id === user.id && !is_read` 的消息
3. 逐条调用 `markMessageRead(msg.id)`

**关键代码**：
- 前端：`frontend/src/pages/PrivateChatPage.tsx:87-99`

#### 已读标记的 REST API 调用链

```
前端 markMessageRead(messageId)
  → apiClient.put(`/api/chat/private/messages/${messageId}/read`)
    → 后端 mark_message_read_endpoint()
      → chat_service.mark_message_read(db, message_id, current_user.id)
        → 查询 PrivateMessage 表
        → 验证消息存在
        → 验证当前用户是接收者（receiver_id == user_id）
        → 更新 is_read = True
        → 更新 read_at = func.now()
        → db.commit()
```

**关键代码**：
- 前端：`frontend/src/services/api/chatService.ts:203-205` — API 调用
- 后端：`backend/app/api/chat.py:408-420` — API 端点
- 后端：`backend/app/services/chat_service.py:653-688` — 已读标记逻辑

### 阶段4：会话列表与未读计数

#### 会话列表加载

1. `ChatHomePage` 组件加载时和每5秒轮询调用 `getPrivateConversations()`
2. REST API：`GET /api/chat/private/conversations`
3. 后端 `chat_service.get_user_conversations()` 逻辑：
   - 查询用户发送和接收过的所有对方用户ID
   - 对每个对方用户，获取最后一条消息（`joinedload` 预加载关系）
   - 统计未读消息数量（`sender_id=对方ID, receiver_id=自己ID, is_read=False`）
   - 按最后消息时间倒序排序
4. 返回会话列表，每项包含：`user_id`、`username`、`last_message`、`unread_count`、`last_message_at`

**关键代码**：
- 前端：`frontend/src/pages/ChatHomePage.tsx:48-55` — 会话加载
- 后端：`backend/app/api/chat.py:324-359` — API 端点
- 后端：`backend/app/services/chat_service.py:576-650` — 会话列表逻辑

#### 未读计数查询

1. 独立接口 `GET /api/chat/private/unread-count?sender_id=X`
2. 可查询总未读数或特定发送者的未读数
3. 用于聊天主页侧边栏显示未读气泡

## 五、关键设计特点

### 1. 双通道通信

| 通道 | 协议 | 用途 |
|------|------|------|
| REST API | HTTP | 拉取历史消息、标记已读、会话列表等 CRUD 操作 |
| WebSocket | WS | 实时消息推送（发消息、收消息） |

### 2. 消息可靠性与离线处理

- **消息持久化优先**：WebSocket 收到消息后先写数据库，再推送。即使接收者不在线，消息也不会丢失
- **离线消息拉取**：用户上线打开私聊页面时，通过 REST API `GET /api/chat/private/{user_id}/messages` 拉取历史消息
- **WebSocket 重连**：前端最多重试5次，间隔3秒，重连成功后恢复心跳

### 3. 发送者回显机制

后端在推送消息时，分别向接收者和发送者各发一次相同的消息体。这确保：
- 发送者能实时看到自己的消息（无需等待 REST 响应）
- 两次推送携带相同的 `id`、`created_at` 等服务端生成的字段，保证一致性

### 4. 已读标记的安全校验

后端 `mark_message_read()` 进行了两层校验：
- 消息必须存在（404）
- 操作者必须是消息的接收者（403），发送者无法标记自己的消息为已读

### 5. 心跳保活

前端每30秒发送 `{type: "ping", timestamp: ...}`，后端回复 `{type: "pong", timestamp: ...}`，防止连接因空闲被关闭。
